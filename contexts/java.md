# Context: Java

## When to use
- `project.language == Java`

## How to apply
- Follow the repo's existing build tool configuration (Maven or Gradle); do not guess.
- When declaring dependencies, **always include required optional/scope qualifiers**. For Maven, verify that optional features used in code have their corresponding dependencies declared (e.g., `spring-boot-starter-validation` for Bean Validation, `spring-boot-starter-data-jpa` for JPA, `jackson-datatype-jsr310` for Java 8 date/time serialization). A missing transitive dependency may compile but fail at runtime with `ClassNotFoundException` or `NoClassDefFoundError`.
- Prefer JUnit 5 for unit/integration tests and Mockito for mocking.
- Keep tests deterministic (no wall-clock dependent assertions; use fixed clocks/mocks when needed).
- **Test database isolation**: use `@Transactional` with rollback (Spring default) or `@DirtiesContext` to ensure each test starts with a clean state. For H2/HSQLDB in-memory databases, prefer `spring.jpa.hibernate.ddl-auto=create-drop` with `@DirtiesContext(classMode=AFTER_EACH_TEST_METHOD)` or use `@Sql` to reset schema. For Testcontainers, always drop/recreate the schema in `@BeforeEach` rather than relying on `@AfterEach` cleanup alone — a crashed test leaves stale data that causes cascading failures. **Never use file-based embedded databases in tests** (e.g., H2 file mode); always use in-memory mode to avoid stale state between runs.
- **Fixture resilience**: test helper methods that create entities via API and extract response fields MUST assert the response succeeded first. Without this, an upstream failure cascades as a confusing NPE or missing-field error across every dependent test.
- Place business logic in service/application layers; keep controllers thin.
- Follow existing exception-to-error mapping patterns so API errors remain consistent.
- **Use domain-specific exceptions** (`NotFoundException`, `DuplicateResourceException`, etc.) at service boundaries. Never catch broad `Exception` or `RuntimeException` in `@ControllerAdvice` without re-throwing unknowns — library code (Jackson, JPA, Spring Security) also throws these base types, and a broad catch silently maps unrelated errors to wrong HTTP status codes.

## Build hygiene
- When the IDE's JDK version differs from the project's configured `java.version`, stale `.class` files compiled by the IDE may have an incompatible class file version. Use `mvn clean test` (or `./gradlew clean test`) to clear the `target/` (or `build/`) directory before running tests. Build-tool `clean` targets only remove build artifacts and are safe to run.
- Always prefer `mvn clean test` over bare `mvn test` to avoid class-file version mismatch errors (e.g., "class file version 65.0, runtime recognizes up to 55.0").

## What not to do
- Do not add new production dependencies without explicit approval.
- Do not modify production code unless the task requires it.
- Do not create tests that assert internal implementation details; assert externally observable behavior.
- Do not hard-code framework-specific HTTP status codes in assertions when the status is generated by the framework (e.g., Spring Security may return 401 or 403 depending on configuration and version). Use range checks for framework-controlled responses; use exact assertions only for application-defined responses.
