# Context: Dotnet

## When to use
- `project.language == Dotnet`

## How to apply
- Follow the repo's existing .NET project layout and build system (`*.csproj`, solution files, CI scripts); do not guess.
- When adding NuGet packages, **verify that all required sub-packages are declared**. Many .NET packages have optional components (e.g., `Microsoft.AspNetCore.Authentication.JwtBearer` for JWT, `Microsoft.EntityFrameworkCore.SqlServer` vs `.Sqlite` for specific providers, `FluentValidation.AspNetCore` for ASP.NET integration). A missing provider package compiles but throws `InvalidOperationException` at runtime. Always match the provider package to what the code actually configures.
- Prefer the repo's current unit test framework (commonly xUnit/NUnit/MSTest) and mocking library (commonly Moq).
- Keep tests deterministic (avoid real network/clock dependencies; use fakes/mocks).
- **Test database isolation**: use `IClassFixture` / `IAsyncLifetime` with `EnsureDeleted()` + `EnsureCreated()` (or `Respawn` library) in setup — not just teardown. For EF Core SQLite, **always use in-memory mode** (`DataSource=:memory:`) with a shared open connection to prevent each context getting its own empty database. For the EF Core in-memory provider, use a unique database name per test (`new DbContextOptionsBuilder().UseInMemoryDatabase(Guid.NewGuid().ToString())`). **Never use file-based SQLite** in tests — stale `.db` files cause cascading false failures between runs. Relying solely on teardown cleanup means a crashed test leaves stale data for the next run.
- **Fixture resilience**: test helper methods that create entities and extract response fields MUST assert the response succeeded first. Without this, an upstream failure cascades as a confusing `NullReferenceException` across every dependent test.
- Place business logic in service layers; keep API/controller layers minimal.
- Apply the repo's existing error handling conventions so API responses are consistent.
- **Use domain-specific exceptions** (`NotFoundException`, `DuplicateResourceException`, etc.) at service boundaries. Never catch broad `Exception` in middleware without re-throwing unknowns — library code (EF Core, System.Text.Json, Identity) also throws base exceptions, and a broad catch silently maps unrelated errors to wrong HTTP status codes.

## What not to do
- Do not add new production dependencies without explicit approval.
- Do not modify production code unless the task explicitly requires it.
- Do not create tests that assert internal state transitions that are not part of the public contract.
- Do not hard-code framework-specific HTTP status codes when the response is generated by middleware (e.g., ASP.NET authentication middleware may return 401 or 403 depending on scheme configuration). Use range checks for framework-controlled responses; use exact assertions only for application-defined responses.
