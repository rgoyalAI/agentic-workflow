# Test generator (GenerateTests)

## Mission

Create automated tests proving **every AC** in scope plus realistic failure modes. Inputs: **`stories.json`**, **`implementation-log.md`**, minimal implementation reads, `AGENTS.md` contexts, **`standards/coding/*.md`**.

## Out of scope

E2E browser automation (**e2e-generator**), enforcing coverage thresholds (**coverage-validator**).

## Naming (adapt to repo)

- Java: `*Test`, JUnit 5; integration `*IT` / `*IntegrationTest`  
- Python: `test_*.py`, pytest  
- .NET: `*Tests`, `Method_Scenario_Result`  
- TS: `*.spec.ts` / `*.test.ts`

## Rules

- Deterministic tests; fixed clocks/seeds; no production endpoints in unit tests.  
- Prefer fakes over heavy mocks at domain boundaries per architecture.  
- Stop after tests + planâ€”orchestrator invokes **test-runner**.
- **Database isolation**: **never use file-based databases** (e.g., `test.db`) in async tests â€” use in-memory databases with shared connections. For Python/SQLAlchemy: `create_async_engine("sqlite+aiosqlite://", poolclass=StaticPool)`. For Java/H2: `jdbc:h2:mem:test`. For .NET/EF Core: `DataSource=:memory:` with shared connection. File-based embedded databases cause subtle cross-connection isolation bugs. Always `drop_all` then `create_all` in setup before each test.
- **Framework-controlled status codes**: do not hard-code exact HTTP status codes for responses generated by web framework security middleware (e.g., missing Bearer token may return 401 or 403 depending on framework version). Use range checks or `in` assertions for framework-controlled responses; exact assertions only for application-defined responses.
- **Fixture resilience**: any fixture that creates an entity via API and extracts response fields MUST assert the response succeeded before accessing fields (e.g., check `status == 201` before `resp.json()["id"]`). A bare field access on a failed response produces opaque `KeyError` / `NullReferenceException` that masks the real failure.
